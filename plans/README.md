## Findings considered and rejected

So nobody re-audits these:

- **`appendEventWithNextSequenceForRun` "duplicate" SQL predicate** (`postgres.ts:1828-1829`): `id = $2 AND id = $8` is a deliberate query-level guard that the event's runId equals the lease guard's runId; removing it would weaken the check. By design.
- **SSE replay "lost events" race** (`app/event-stream.ts:144-172`): events are durable in Postgres and reconnect replays from the cursor; the live buffer only defers delivery, never loses it. High-water cutoff (`cleanup(true)`) forces a clean reconnect. By design.
- **Preview token issued with `canReadSession`** (`server.ts:1437`): not a privilege escalation — the proxy re-authorizes every request and gates unsafe methods on `canWrite` (`service-proxy.ts:237,798,844-846`). By design.
- **Preview cookie cross-port reuse / unbounded renewal**: tokens are port-bound at validation (`service-proxy.ts:788`) and renewals are capped at the immutable `grantExp` set at issuance (`:289,810,820`). By design.
- **Generic webhook "missing auth"**: bearer auth is enforced in the service (`generic-webhook/service.ts:42-44,101-103`); only the comparison strategy needed fixing (plan 002).
- **Callback SSRF**: `parseHttpCallbackUrl` + `resolveSafeCallbackAddresses` already block credentials, non-http(s) schemes, local hostnames, and private IPs (`callbacks/service.ts:293-330`). Solid as-is.
- **CORS misconfiguration**: allowlist is exact (localhost dev origins + configured `webBaseUrl`); `sec-fetch-site`/origin checks cover cookie-auth CSRF. By design.
- **Web session-switch state races** (`app.tsx`): flush/apply paths all check `selectedSessionIdRef.current` against the response's sessionId before applying (`app.tsx:1066,1305,1332`); the in-flight coalescing is sound under single-threaded JS. Defensive code already present.
- **WebSocket pipe cleanup in sandbox-bridge** (`server.ts:212-232`): destroy-both-on-error is the standard pattern; double-destroy is a no-op in Node. Not worth churn. (The HTTP-path `headersSent` issue in the same file IS real — plan 008.)
- **Web E2E "not in CI"**: false — `web-e2e` job exists in `run-tests.yml:111` (the `if:` only skips release-please commits, same as every job).
- **Orchestrator server duplication** (`docker-orchestrator-server.ts` vs `k8s-agent-sandbox-orchestrator-server.ts`): two ~65-line bootstraps with mostly different env/config; ~20 truly shared lines. Below the consolidation threshold.
- **Sandbox-secrets insert N+1** (`postgres.ts:1510,1568`) and **stale-recovery query batching**: real patterns, negligible real cost (rare operations dominated by provider latency). Not worth doing.
- **Shiki bundling**: already behind a cached dynamic `import('shiki')` (`highlighted-code.tsx:136`); Vite splits dynamic imports into separate chunks by default. Would need bundle-analysis evidence to reopen.
- **Lease-renewal clock-skew boundary**: millisecond-window theoretical; heartbeats give many renewal chances. Not actionable.
- **Rate limiting on authenticated routes**: legitimate hardening for a later maturity stage; all state-changing surfaces already require auth, and the deployment is single-operator today. Revisit when multi-tenant.
- **Events-table retention / full-type compaction**: explicitly deferred by maintainers as a product decision (commit history for #59/#61); re-raising it is noise.
- **`@earendil-works/pi-*` exact pins**: plausibly deliberate (API churn in 0.7x); documentation nit at most.
- **dist/ `.d.ts` referencing `@flue/sdk`**: build-artifact drift only; source uses `@flue/runtime`.
